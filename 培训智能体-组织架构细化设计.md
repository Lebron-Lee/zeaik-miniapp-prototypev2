# 培训智能体 · 组织架构与无感注册机制细化设计

> **版本**：v1.1 · **日期**：2026-04-12 · **关联文档**：培训智能体产品设计方案 v1.0

---

## 一、核心问题：同层级多角色的本质矛盾

原方案采用纯金字塔「层级」模型，每一层只有一种角色（如区域经理）。但真实餐饮企业的组织结构中，**同一层级往往存在多种职能角色**，例如：

> 区域层级 = 区域经理（线务管理）+ 数据管理员（数据权限）+ 经理助理（协同执行）+ 人力资源经理（人员管理）

这些角色在**行政层级上平级**，但在**数据权限、培训发起权、查看范围**上存在本质差异。若仍用单一层级描述，会导致：
- 数据管理员无法查看员工培训数据（权限不足）
- 人力资源经理无法发起全区域培训（功能受限）
- 经理助理误收到不该接收的管理看板（权限过宽）

因此，需要在「层级」之上引入「**角色类型（Role Type）**」维度，形成**层级 × 角色**的二维组织模型。

## 二、组织模型重新定义

### 2.1 双维度组织模型

系统用两个独立维度描述每个用户的身份：

| 维度 | 含义 | 示例值 |
|---|---|---|
| **层级（Level）** | 在组织金字塔中的纵向位置 | 集团级 / 区域级 / 门店级 / 班组级 / 员工级 |
| **角色类型（Role Type）** | 在该层级中承担的职能 | 管理者 / 职能支持 / 数据观察者 / 执行者 |

每个用户的完整身份 = **层级 + 角色类型**，例如：

- 区域经理 = **区域级** + **管理者**
- 区域数据管理员 = **区域级** + **数据观察者**
- 区域人力资源经理 = **区域级** + **职能支持（人力）**
- 店长 = **门店级** + **管理者**
- 服务员 = **员工级** + **执行者**

### 2.2 角色类型权限矩阵

不同角色类型在同一层级内的权限差异如下：

| 权限项 | 管理者 | 职能支持 | 数据观察者 | 执行者 |
|---|:---:|:---:|:---:|:---:|
| 创建培训任务 | ✅ | ✅（限本职能域） | ❌ | ❌ |
| 向下级发起培训 | ✅（全部下级） | ✅（本职能相关下级） | ❌ | ❌ |
| 查看下级培训数据 | ✅（全部） | ✅（本职能域） | ✅（只读全部） | ❌ |
| 接受上级培训任务 | ✅ | ✅ | ✅ | ✅ |
| 查看问题清单看板 | ✅ | ✅（本职能） | ✅（只读） | ❌ |
| 管理题库 | ✅ | ✅（本职能域） | ❌ | ❌ |
| 查看员工情绪报告 | ✅ | ✅（人力类） | ❌ | ❌ |

**说明**：
- **管理者**：对下级拥有完整管理权，是培训任务的主要发起者（区域经理、店长、班组长）
- **职能支持**：在本职能域内有发起和查看权限（人力资源经理可发起人员类培训；数据管理员可查看数据但不发起培训）
- **数据观察者**：只读权限，可查看所有数据但不能操作（适合数据分析、审计角色）
- **执行者**：只能接受和完成培训任务，无管理功能

## 三、同层级多角色的注册与绑定设计

### 3.1 注册流程中的角色选择

员工通过培训任务链接进入注册时，需完成三步身份绑定：

```
第一步：填写基本信息
  手机号 + 姓名 + 验证码

第二步：选择所属层级
  [集团级] [区域级] [门店级] [班组级] [员工级]
  ↓ 选择后展示该层级的常见角色类型供选择

第三步：选择角色类型（同层级内）
  以「区域级」为例，展示：
  ○ 区域经理（管理者）
  ○ 区域助理（职能支持）
  ○ 人力资源经理（职能支持-人力）
  ○ 数据管理员（数据观察者）
  ○ 其他（自定义填写职位名称）

第四步：选择直属上级
  搜索/选择上级姓名（支持多选，允许多个上级）
```

**关键设计原则**：
- 角色类型由用户自选，但系统会根据选择**自动映射权限组**，无需管理员手动配置
- 「其他」选项允许自定义职位名称，系统默认归入「执行者」权限组，后续由管理者手动升级权限
- 注册完成后，用户的角色类型可由其直属上级（管理者）在后台修改

### 3.2 多上级绑定机制

同层级多角色场景中，一个员工可能需要向多个上级汇报（矩阵管理）。系统支持**一对多上级绑定**，规则如下：

| 场景 | 处理方式 |
|---|---|
| 区域助理同时汇报给区域经理和集团人力总监 | 注册时可选择多个上级，培训任务来自任意上级均可接收 |
| 培训结果可见范围 | 所有已绑定的上级均可查看该员工的培训数据 |
| 任务冲突（两个上级同时发起同类培训） | 系统提示员工「有重复培训任务」，员工可选择合并完成或分别完成 |
| 上级变更 | 员工可申请变更上级，需原上级或更高层级管理者审批 |

### 3.3 无感注册的链接参数设计

培训任务分享链接需携带足够的上下文参数，使新用户注册时能自动预填部分信息，减少操作步骤：

```
链接结构示例：
https://miniapp.zeaik.com/training?
  task_id=T20260412001        ← 培训任务ID
  &inviter_id=U10086          ← 发起人ID（自动成为候选上级）
  &inviter_level=2            ← 发起人层级（引导新用户选择层级）
  &org_id=ORG001              ← 组织ID（自动归属到该组织）
  &role_hint=staff            ← 角色提示（发起人预设的接收者角色）
```

新用户点击链接后的预填效果：
- **所属组织**：自动填入（来自 `org_id`）
- **直属上级候选**：发起人自动出现在上级候选列表首位（来自 `inviter_id`）
- **层级建议**：根据 `inviter_level` 推断新用户可能的层级（发起人是区域级管理者，则建议新用户选「门店级」或「员工级」）
- **角色提示**：若发起人预设了接收者角色（如「服务员」），则在角色选择页高亮该选项

## 四、组织架构的自动构建逻辑

### 4.1 从培训传播到组织图谱

无感注册的核心价值在于：**每一次培训任务的传播，都是一次组织架构的自动填充**。其传播路径如下：

```
集团总部发起培训
      │ 分享链接（携带 org_id + inviter_id）
      ▼
区域经理接收 → 注册时绑定「集团总部」为上级
      │ 区域经理再次分享给下属
      ▼
店长接收 → 注册时绑定「区域经理」为上级
      │ 店长分享给全店员工
      ▼
员工接收 → 注册时绑定「店长」为上级
```

经过一次完整的培训传播，系统自动沉淀出：
- 完整的四级组织架构（集团 → 区域 → 门店 → 员工）
- 每个节点的角色类型和权限
- 上下级关系图谱（支持多上级的矩阵结构）

### 4.2 同层级多角色的组织图谱展示

在管理后台的组织架构图中，同层级多角色以**横向展开**方式呈现，与纵向层级明确区分：

```
                    ┌─────────────────────────────────────────┐
                    │            集团总部（集团级·管理者）        │
                    └─────────────────┬───────────────────────┘
                                      │
          ┌───────────────────────────┼───────────────────────────┐
          │                           │                           │
┌─────────┴──────────┐   ┌────────────┴───────────┐   ┌──────────┴──────────┐
│ 华北区域经理         │   │ 华东区域经理              │   │ 华南区域经理          │
│ （区域级·管理者）    │   │ （区域级·管理者）         │   │ （区域级·管理者）     │
│                    │   │                         │   │                     │
│ ┌────────────────┐ │   │ ← 同层级横向展开 →       │   │                     │
│ │ 区域数据管理员  │ │   │                         │   │                     │
│ │（区域级·观察者）│ │   │                         │   │                     │
│ ├────────────────┤ │   │                         │   │                     │
│ │ 区域人力经理   │ │   │                         │   │                     │
│ │（区域级·职能）  │ │   │                         │   │                     │
│ └────────────────┘ │   │                         │   │                     │
└─────────┬──────────┘   └────────────┬───────────┘   └──────────┬──────────┘
          │                           │                           │
       门店层级...                  门店层级...                  门店层级...
```

**设计原则**：
- 纵向代表层级（上下级关系）
- 横向代表同层级内的不同角色（平级关系）
- 管理者角色始终在同层级的最左侧（主节点）
- 职能支持和数据观察者以附属卡片形式挂靠在管理者节点旁

## 五、注册引导的 UX 设计要点

### 5.1 降低注册摩擦的关键策略

同层级多角色的引入增加了注册时的选择复杂度，需通过以下 UX 策略降低摩擦：

**策略一：智能角色推荐**。根据链接携带的 `role_hint` 参数和发起人层级，系统在角色选择页自动高亮最可能的角色，用户只需确认而非从头选择。

**策略二：角色描述简化**。不展示「管理者/职能支持/数据观察者」等系统术语，而是展示用户熟悉的职位名称，系统在后台完成职位名称到权限组的映射：

| 用户看到的选项 | 系统内部映射 |
|---|---|
| 店长 | 门店级 · 管理者 |
| 副店长 | 门店级 · 管理者（副） |
| 收银员 | 门店级 · 执行者 |
| 厨师长 | 门店级 · 职能支持（厨务） |
| 服务员 | 员工级 · 执行者 |
| 区域经理 | 区域级 · 管理者 |
| 人力专员 | 区域级 · 职能支持（人力） |

**策略三：先完成培训再完善资料**。注册流程采用「最小必填」原则，仅要求手机号+姓名+直属上级即可完成注册并开始答题。层级、角色类型等信息可在首次培训完成后引导补充，或由上级在后台代为设置。

**策略四：上级审核确认**。新用户注册绑定上级后，系统向上级发送「新成员加入」通知，上级可确认或修改该成员的角色类型，形成双向确认机制，避免角色误选导致的权限问题。

### 5.2 注册页面流程图

```
用户点击培训链接
      │
      ▼
是否已注册？
  ├── 是 → 直接进入培训任务
  └── 否 → 进入注册流程
              │
              ▼
         手机号验证（30秒）
              │
              ▼
         填写姓名
              │
              ▼
         选择职位（下拉列表，含「其他」）
         系统自动推荐：根据链接参数高亮候选项
              │
              ▼
         选择直属上级
         系统自动推荐：发起人排在首位
         支持搜索姓名/手机号
         支持多选（矩阵管理场景）
              │
              ▼
         注册完成 → 向上级发送「新成员加入」通知
              │
              ▼
         直接进入培训任务（无需等待上级审核）
```

## 六、待确认事项（新增）

在原方案 6 个待确认事项基础上，补充以下问题：

7. **职位名称标准化**：系统内置的职位名称列表由谁维护？是否允许企业自定义职位名称？建议由集团管理员在后台维护本企业的职位名称表，系统提供默认餐饮行业职位模板。

8. **多上级的任务优先级**：当员工有多个上级同时发起培训时，任务优先级如何排序？建议按发起时间排序，员工可自行调整完成顺序。

9. **角色变更的历史数据归属**：员工从「执行者」升级为「管理者」后，其历史培训数据如何处理？建议历史数据保留在原层级，新权限仅对变更后的数据生效。

10. **组织架构图的可见范围**：员工是否能看到完整的组织架构图？建议员工只能看到自己及直属上级，管理者可看到本层级以下的完整架构。
